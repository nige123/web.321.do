# 321 gobin — Design

**Goal:** Extend `321` with a `gobin` command that builds, signs, and
releases cross-architecture Go binaries (starting with the `123` CLI)
from the box, so distribution and safe auto-update ride the same
manifest-driven tooling as service deploys.

## Context

`321` is the ecosystem's control surface for *running services*
(`321 go`, `321 restart`, …), driven by `321.yml` manifests. The `123`
CLI/TUI now needs to ship as a self-updating binary across
linux/darwin/windows × amd64/arm64. The chosen architecture (agreed
separately):

- The **website** (123.do) is the front door — it serves the install
  script, a download page, and a `version.json` manifest. *Separate spec.*
- The **bytes** live in **S3** (+ CDN); the website/API redirect to them.
- The **API** enforces a `min_supported` client version via a request
  header. *Separate spec.*
- The **`123` binary** self-updates by polling `version.json`, verifying
  an ed25519 signature against a baked-in public key, self-checking, and
  applying on next launch. *Separate spec (consumer, in tui.123.do).*

This design covers only the piece that **produces** `version.json` and
the signed artifacts: the `321 gobin` command.

## Scope

**In:** the `321 gobin make | release | rollback | status` command; the
`gobin:` block in a repo's `321.yml`; the `version.json` format; ed25519
signing; generated-vs-checked-in GoReleaser config.

**Out (separate specs, downstream consumers of `version.json`):** the
website install/download/manifest routes; the API `min_supported`
handshake; the self-update client baked into the `123` binary.

## Command surface

Two honest verbs — build, then publish — so building without publishing
is natural, and "releasing"/rolling back are pure manifest-pointer flips.

### `321 gobin make [--bump patch|minor|major | --version X.Y.Z] [--min-supported X.Y.Z]`

Creates all the binaries. Steps:
1. **Resolve version.** `--bump` computes the next semver from the last
   released version (read from `version.json`); `--version` overrides.
2. **Tag.** Create and push the annotated git tag `v<version>` (321
   automates tagging).
3. **Config.** Use a checked-in `.goreleaser.yaml` if present; otherwise
   generate one from the `gobin:` block.
4. **Build + sign.** Run GoReleaser: cross-compile the target matrix
   with `CGO_ENABLED=0`, stamp `<version>` via ldflags into
   `gobin.version_var`, produce archives + `SHA256SUMS`, and sign each
   artifact with the ed25519 key.

Output: artifacts in `dist/` on the box, **plus a `dist/gobin-meta.json`**
recording the version, resolved `min_supported` (defaults to the previous
release's), and per-arch checksums. That file is the handoff `release`
reads — since `make` and `release` are separate invocations, the version
and floor pass between them on disk, not in memory. Nothing is uploaded
or made live.

### `321 gobin release [version]`

Releases them. `version` defaults to the one in `dist/gobin-meta.json`
(the last `make`). Steps:
1. **Upload** the built artifacts to S3 under immutable, version-scoped
   paths (`<prefix>/<version>/<name>_<os>_<arch>.tar.gz` + `.sig` +
   checksum).
2. **Write the manifest.** Read the current `version.json` from S3; add
   this version's per-arch `{url, sha256, sig}` under `builds` (urls +
   checksums from the upload, `min_supported` from `gobin-meta.json`);
   set `latest` → this version; prune builds older than the retained
   window. Write it back.
3. **Verify.** Re-fetch the manifest, `HEAD` each per-arch URL (200), and
   confirm each `.sig` object resolves. Report what is now live.

Errors clearly ("nothing built for `<version>` — run `321 gobin make`")
when the artifacts aren't in `dist/`.

### `321 gobin rollback`

Flip `version.json`'s `latest` back to the previous entry in `builds`
(whose bytes are still in S3) and write it back. Instant, byte-free — the
kill-switch. Refuses if there is no prior build to fall back to.

### `321 gobin status`

Print, per arch: the version built locally in `dist/` (if any) and the
`latest` from the live manifest, so it's obvious what would be released
and what's currently live.

## Data

### The `gobin:` block (in the repo's `321.yml`)

Read by the existing `Deploy::Manifest`, alongside service config.

```yaml
gobin:
  name: 123                    # binary + artifact base name
  main: .                      # Go package to build
  targets:                     # optional; default = the five below
    - linux/amd64
    - linux/arm64
    - darwin/arm64
    - darwin/amd64
    - windows/amd64
  version_var: main.version    # ldflag target for the stamped version
  s3:
    bucket: 123do-releases
    prefix: bin/123
  sign_key: gobin_signing_key  # key name in conf/secrets.conf (private ed25519)
  retain: 5                    # optional; how many past builds to keep in the manifest
```

A checked-in `.goreleaser.yaml` in the repo overrides generation (escape
hatch); the `gobin:` block still supplies `s3`, `sign_key`, and `retain`.

### The `version.json` manifest (in S3; served by the website)

```json
{
  "name": "123",
  "latest": "1.4.0",
  "min_supported": "1.2.0",
  "builds": {
    "1.4.0": {
      "darwin/arm64": { "url": "…", "sha256": "…", "sig": "…" },
      "linux/amd64":  { "url": "…", "sha256": "…", "sig": "…" }
    },
    "1.3.0": { "…": {} }
  }
}
```

`latest` indexes into `builds`; `rollback` re-points it; entries beyond
`retain` are pruned (their S3 bytes may be garbage-collected separately —
out of scope). The self-updater reads `latest`/`min_supported`, picks its
arch, downloads the `url`, and verifies `sha256` + `sig`.

## Trust & signing

ed25519. The **private** key lives in `conf/secrets.conf` (gitignored,
the existing per-host secrets pattern) under `gobin_signing_key`; 321
passes it to the signer and never logs it. The **public** key is embedded
in the `123` binary so the self-updater verifies downloads against it.
The signature — not the transport or any auth — is the trust anchor, so
anonymous public downloads from the website/S3 are safe.

## Components (isolation)

- **`Deploy::Command::gobin`** — arg parsing, subcommand dispatch, and
  user-facing output/preflight messaging. Thin.
- **`Deploy::GoBin`** — the pipeline library the command drives: version
  resolution, tagging, `.goreleaser.yaml` generation, GoReleaser
  invocation, S3 upload, and manifest read/modify/write (including the
  `rollback` re-point and pruning). Each of these is a small, separately
  testable method with GoReleaser and S3 behind seams.
- **`Deploy::Manifest`** — extended to surface the `gobin:` block.

The GoReleaser call and the S3 client are the two external boundaries;
everything else (version math, config generation, manifest mutation) is
pure and unit-testable without touching the network or the toolchain.

## Error handling / preflight (fail before acting)

Mirroring `321 go`'s preflight:

- **`make`:** clean git tree; resolved version is valid semver and newer
  than `latest`; `go` and `goreleaser` on PATH; `sign_key` present in
  secrets. Any failure aborts *before* tagging or building.
- **`release`:** `dist/` holds artifacts for the target version; S3 creds
  present; the manifest object is readable/writable. The verify step at
  the end is itself a guard — a failed `HEAD`/signature check reports a
  broken release rather than a silent success.
- **`rollback`:** a prior build exists in `builds`.

No verb leaves a half-state: tagging happens only after preflight passes;
the manifest is written once, last, after uploads succeed.

## Testing

- **Unit (pure logic):** version resolution (`--bump` math, "must be
  newer than latest"), `.goreleaser.yaml` generation from a `gobin:`
  block, `version.json` mutation (add build, set latest, prune to
  `retain`), and `rollback` re-pointing — all with GoReleaser and S3
  stubbed.
- **Boundary seams:** inject a fake GoReleaser runner (asserts the
  generated config + captured args) and a fake S3 client (records
  puts/gets), so `make`/`release`/`rollback` are drivable end-to-end in
  tests without the toolchain or network.
- **Integration (optional):** a `--dry-run` that builds for the host arch
  only and writes the manifest to a local path, plus a minio-backed run
  for real S3 round-trips, gated like the existing live tests.

## Out of scope

- The website routes, the API `min_supported` handshake, and the `123`
  self-update client — each its own spec, all downstream of the
  `version.json` format defined here.
- S3 lifecycle/GC of pruned builds; CDN configuration.
- A combined `make`+`release` shortcut (deliberately two verbs; add later
  only if the two-step proves annoying in practice).
